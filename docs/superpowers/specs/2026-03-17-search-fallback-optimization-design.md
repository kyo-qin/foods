# 多类型图片抓取搜索回退优化设计

## 目标

提升 8 个预设类型的抓图成功率，优先改善 `beverage`、`milk_tea`、`coffee` 三类饮品相关类型。优化范围限定在“搜索候选图生成”层，不调整下载、图片标准化、存储和清单导出逻辑。

## 问题现状

当前程序每个词条只使用一个固定搜索后缀，例如 `beverage` 只会查询一次：

```text
可乐 饮料 酒水 实拍 产品图
```

这种强限定词在饮品、奶茶、咖啡、打包盒等类型上容易导致：

- 某些词条完全无搜索结果
- 搜索结果数量过少，无法补齐到后续下载和过滤阶段
- 命中质量随类型变化明显，统一模板适配性差

实际验证中，`python3 -m src.main --type beverage` 能正常处理 20 个词条，但最终写出 0 张图片，说明当前强限定单查询策略对饮品类成功率不足。

## 本期范围

### 包含

- 将单个 `query_suffix` 升级为按类型配置的 `query_suffixes`
- 每个词条按查询列表顺序执行搜索回退
- 对多轮查询结果按 URL 去重并合并
- 达到候选上限时提前停止后续查询
- 记录每层查询命中数量的日志，便于继续调优

### 不包含

- 新增搜索引擎
- 调整下载重试或请求头策略
- 调整图片过滤规则
- 引入图像理解模型
- 新增调试文件输出格式

## 优化方案比较

### 方案 A：单查询放宽

把当前单个查询直接改成更宽松的模板。

优点：

- 改动最小

缺点：

- 失去高精度查询的价值
- 容易让家常菜等类型引入更多无关图

### 方案 B：查询回退链

每个类型配置多个后缀模板，按顺序尝试，结果去重合并。

优点：

- 能先保留高精度查询
- 失败时再逐层放宽，兼顾质量和成功率
- 改动集中，结构清晰

缺点：

- 请求量比单查询更高

### 方案 C：按类型自定义 provider

不同类型走不同搜索 provider 策略。

优点：

- 上限更高

缺点：

- 第一轮优化过重
- 超出当前问题最短解决路径

### 结论

采用方案 B，并保留少量类型定制。即：统一实现多查询回退框架，由每个类型提供自己的查询后缀列表。

## 配置设计

`src/categories.py` 中的 `CategoryConfig` 从：

- `query_suffix: str`

升级为：

- `query_suffixes: tuple[str, ...]`

示例：

```python
CategoryConfig(
    type_key="beverage",
    ...,
    query_suffixes=(
        "饮料 酒水 实拍 产品图",
        "饮料 实拍",
        "饮料",
        "",
    ),
)
```

## 查询策略

### 执行规则

对每个词条：

1. 按 `query_suffixes` 顺序构造查询
2. 每轮调用搜索 provider 获取候选图
3. 以 `source_url` 为键去重
4. 将新结果追加到候选集合
5. 若累计候选数达到 `search_candidate_limit`，立即停止
6. 若当前查询无结果，则继续下一个更宽松查询
7. 若当前查询有结果但累计仍不足上限，也继续补下一个查询

### 停止条件

- 候选数量达到上限
- 类型配置的查询列表全部耗尽

## 搜索层改造

### 统一搜索接口

保持搜索 provider 接受单个查询字符串，避免把“回退逻辑”塞进每个具体 provider。

建议做法：

- `build_query(item_name: str, query_suffix: str) -> str`
- 在 `FallbackSearchProvider` 外层新增一个小的多查询协调器，或直接在 `src.search.py` 中新增辅助函数

职责分工：

- provider：执行一次具体查询
- 多查询协调器：控制查询回退、去重和停止条件

这样职责清晰，也不会把 DuckDuckGo/Wikimedia 的实现耦合到类别策略里。

## 日志设计

为每个词条输出每层查询的简明日志，例如：

```text
INFO category=beverage item=可乐 query="可乐 饮料 酒水 实拍 产品图" candidates=0
INFO category=beverage item=可乐 query="可乐 饮料 实拍" candidates=4
INFO category=beverage item=可乐 unique_candidates=4
```

目标：

- 能快速看出某个词条在哪一层开始有结果
- 能判断是“完全搜不到”还是“第一层太严”
- 为后续继续调模板提供依据

## 测试设计

至少增加以下测试：

### 配置测试

- 每个类型配置包含多个查询模板
- 饮品类模板包含空后缀回退

### 搜索策略测试

- 多个查询返回重复 URL 时只保留一份
- 第一层无结果时会继续下一层
- 前几层累计达到 `search_candidate_limit` 后停止
- 空后缀时查询退化为原词条

### 回归测试

- 现有 pipeline 行为不变
- manifest 和输出目录结构不受影响

## 风险与取舍

- 回退到空后缀会提高无关图比例，但对饮品类成功率提升明显，第一轮接受
- 请求数增加会让任务变慢，但每个词条在达到候选上限后会尽早停止，可控
- 如果日志太多，后续可再增加 `DEBUG` 级别控制；本期先保证可见性

## 交付结果

完成后，程序应能：

- 为每个类型按顺序尝试多个查询模板
- 将结果去重并补足到候选上限
- 优先提升饮品相关类型成功率
- 在日志中清楚显示每层查询的命中情况
