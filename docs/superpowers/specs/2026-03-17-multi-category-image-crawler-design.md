# 多类型图片抓取扩展设计

## 目标

将现有仅面向中餐菜品的图片抓取工具扩展为“预设类型驱动”的通用图片抓取工具。第一版内置 8 个类型：

- `home_cooking`
- `beverage`
- `takeout_box`
- `staple_food`
- `dessert`
- `noodle`
- `milk_tea`
- `coffee`

每个类型维护 20-30 个常用词条，每个词条默认抓取 3 张图片。用户可通过命令行选择单个类型执行抓取任务。

## 本期范围

### 包含

- 将“dish/dishes”语义抽象为“category/items”
- 增加类型注册表，集中维护每个类型的词表路径、搜索词模板、输出目录和抓取参数
- 新增命令行参数 `--type`
- 按类型拆分数据文件
- 按类型隔离输出目录
- 为类型选择、配置解析、主流程接线补充测试

### 不包含

- 一次运行多个类型的批处理模式
- 图形界面
- 新增搜索引擎或站点专用适配器
- 基于图像模型的内容识别筛图
- 自动去重、相似图识别或版权检测

## 用户需求摘要

- 预设 8 个图片抓取类型
- 每个类型准备 20-30 个常用词条
- 每个词条抓取 3 张图片
- 重点支持中国大陆常用饮料等场景
- 保持现有程序可扩展，后续新增类型时尽量只加配置和词表

## 现状问题

当前工程虽然搜索词列表可以手动替换，但整体命名、配置和入口仍然绑定在“菜品”语义上：

- `src/main.py` 写死读取 `data/dishes.txt`
- `src/pipeline.py` 的返回结构和变量命名均使用 `dish`
- README 说明以菜品为中心
- 搜索词模板默认偏向“中餐 菜品图 实拍”

这使得“饮料、打包盒、奶茶、咖啡”等类型虽然勉强可用，但使用和扩展方式不稳定，容易导致配置散落在代码中。

## 设计原则

- 类型配置中心化，避免新增一个类型就改多处业务代码
- 运行入口统一，只通过 `--type` 切换内容
- 对现有下载、处理、存储、清单导出流程尽量少改
- 保持每个词条抓 3 张的既有行为不变
- 新增类型时，优先只新增词表和注册表配置

## 方案选择

### 方案 A：最小改造

将 `data/dishes.txt` 拆成多个 txt 文件，并在 `main` 中用 `if/elif` 按类型切换。

优点：

- 改动最小
- 上手快

缺点：

- 类型配置分散
- 新增类型需要继续修改主流程代码
- 不利于维护搜索词模板和输出结构

### 方案 B：配置中心化

增加类型注册表，定义每个类型的元数据，主流程只依赖“选中的类型配置”运行。

优点：

- 类型扩展稳定
- 配置集中，逻辑清晰
- 后续可自然扩展到 `--all` 或自定义类型

缺点：

- 需要一次性重命名一部分“dish”语义

### 方案 C：类型插件化

每个类型独立成 handler/provider。

优点：

- 灵活性最高

缺点：

- 对当前项目过度设计
- 会增加不必要的抽象层

### 结论

采用方案 B。它能覆盖当前需求，同时保留后续扩展空间，复杂度也与现有项目规模匹配。

## 数据与目录设计

### 数据目录

新增按类型拆分的词表目录：

```text
data/
└── categories/
    ├── home_cooking.txt
    ├── beverage.txt
    ├── takeout_box.txt
    ├── staple_food.txt
    ├── dessert.txt
    ├── noodle.txt
    ├── milk_tea.txt
    └── coffee.txt
```

每个文件包含 20-30 个词条，一行一个。

### 输出目录

输出路径按类型和词条两级组织：

```text
output/
└── images/
    └── beverage/
        ├── 可乐/
        │   ├── 001.jpg
        │   ├── 002.jpg
        │   └── 003.jpg
        └── 椰子水/
            ├── 001.jpg
            ├── 002.jpg
            └── 003.jpg
```

这样可以避免不同类型下同名词条互相覆盖，也更利于后续批量导入与检查。

## 配置模型设计

新增一个类型配置对象，负责描述单个预设类型的运行元数据。建议字段：

- `type_key`：类型键，如 `beverage`
- `label`：面向人的中文名，如“酒水饮料”
- `items_file`：词表文件路径
- `query_suffix`：搜索附加词
- `per_item_limit`：每个词条保留图片数，默认 3
- `search_candidate_limit`：候选图片数量，默认沿用现有值

在配置模块中维护一个注册表：

```python
CATEGORY_CONFIGS = {
    "beverage": CategoryConfig(...),
    "coffee": CategoryConfig(...),
}
```

并提供统一解析入口，例如：

- `get_category_config(type_key: str) -> CategoryConfig`
- `list_category_types() -> list[str]`

## 搜索词策略

现有 `build_query()` 偏向中餐菜品，需要改成基于类型配置拼接搜索词。新规则：

```python
def build_query(item_name: str, query_suffix: str) -> str:
    return f"{item_name} {query_suffix}".strip()
```

第一版推荐模板：

- `home_cooking`: `家常菜 实拍 菜品图`
- `beverage`: `饮料 酒水 实拍 产品图`
- `takeout_box`: `打包盒 外卖餐盒 实拍`
- `staple_food`: `主食 实拍 菜品图`
- `dessert`: `甜品 实拍 高清`
- `noodle`: `面食 实拍 菜品图`
- `milk_tea`: `奶茶 饮品 实拍`
- `coffee`: `咖啡 饮品 实拍`

运行时的实际查询为 `词条 + 类型后缀`，例如：

- `可乐 饮料 酒水 实拍 产品图`
- `美式咖啡 咖啡 饮品 实拍`
- `铝箔餐盒 打包盒 外卖餐盒 实拍`

## 运行入口设计

主程序新增命令行参数：

```bash
python -m src.main --type beverage
```

行为规则：

- `--type` 为必填或提供合理默认值；第一版建议保留默认值 `home_cooking` 以兼容当前使用习惯
- 启动时解析类型配置
- 读取该类型对应的词表文件
- 输出目录设为 `output/images/<type_key>/`
- `manifest` 文件暂时继续输出到 `output/manifest.csv`

若后续需要支持多类型连续执行，再增加 `--all` 或 `--types`，本期不做。

## 主流程改造

### 入口层

`src/main.py` 不再直接读取 `data/dishes.txt`，而是：

1. 解析 `--type`
2. 读取对应 `CategoryConfig`
3. 载入该类型词条
4. 构造搜索 provider
5. 将“类型配置”和“词条列表”传入 pipeline

### 流水线层

`src/pipeline.py` 需要从“菜品”语义泛化为“词条”语义，但不改变核心行为：

- `process_dish` 重命名为更通用的函数，如 `process_item`
- `DishProcessResult` 重命名为通用结果类型，如 `ItemProcessResult`
- 结果字段中的 `dish_name` 改为 `item_name`

核心逻辑保持：

- 搜索候选图
- 下载
- 标准化处理
- 保存前 3 张有效结果
- 记录清单行

### 清单层

`manifest.csv` 建议新增 `category_type` 字段，用于区分来源类型。原有 `dish_name` 字段建议升级为更通用的 `item_name`。

建议字段：

- `image_id`
- `category_type`
- `item_name`
- `file_path`
- `width`
- `height`
- `source_url`
- `status`

若本期为了兼容现有测试和下游流程，不能一次改字段，也可以保留旧字段并新增 `category_type`。推荐优先做通用命名，避免后续继续受限。

## 容错策略

- 单张图片下载失败：跳过，继续当前词条
- 单个词条最终不足 3 张：记录缺失数量，任务继续
- 词表文件不存在：直接报错退出
- 传入未知 `--type`：直接报错退出，并输出支持的类型列表
- 单个搜索 provider 失败：维持现有 fallback 逻辑

目标是保证批量任务在局部失败时仍可完成，同时在输入错误时尽早失败。

## 风险与取舍

### 命名兼容性

将 `dish_name` 改为 `item_name` 会波及若干测试和清单结构，但这是必要的基础性修正。若继续保留菜品语义，后续每扩展一次类型都会增加理解成本。

### 搜索质量波动

“打包盒”等非食品类关键词可能比菜品更容易搜到广告图或电商图。第一版接受这一现实，主要依赖搜索后缀和候选数量提升结果质量，不引入复杂筛选逻辑。

### manifest 兼容

如已有外部脚本依赖旧字段，需要评估是否保留兼容字段。当前项目内若无下游依赖，推荐直接升级为通用字段。

## 交付结果

完成后，工程应能：

- 通过 `--type` 选择 8 个预设类型之一执行抓取
- 每个类型使用独立词表和搜索模板
- 输出按 `类型/词条` 分层组织的图片目录
- 为每个词条保留 3 张图片
- 输出包含类型信息的 `manifest.csv`
- 通过新增和调整后的自动化测试
